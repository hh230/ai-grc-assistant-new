# ADR 0040: Rasheed V2 — the tenancy model (the contract, decided before it is built)

- Status: Accepted — contract only; implementation deferred (Phase 15+)
- Date: 2026-07-16
- Deciders: Product Owner (via approved Phase 14.5 stabilization task), Architecture
- Related: CLAUDE.md §3, §7, §8, §9, §12, §16, §19, §20; ADR 0014 (security &
  multi-tenancy — the principle this ADR makes concrete for V2), ADR 0015 (audit &
  traceability), ADR 0035 (knowledge library — the two-library scope), ADR 0038
  (pipeline-contracts + ai-orchestrator), ADR 0039 (platform hardening)

## Context

CLAUDE.md §20 has said from day one that tenant isolation is absolute and structural —
"impossible by construction, not by convention." ADR 0014 accepted that as a principle. The
V2 pipeline does not implement it. Today, in `v2/packages`:

- no contract carries a tenant: not `UserRequest`, not `RetrievalQuery`, not `Filter`, not
  `DecisionPlan`, not `ContextPackage`, not `LLMRequest`, not `AuditRecord`, not any
  `DomainEvent`;
- `Filter` carries a comment — "Tenant scope will slot in here when the artifacts carry
  it" — which is the entire extent of the mechanism;
- the retrieval architecture doc already specifies the intended rule (`scope_kind =
  'global'` ∪ (`scope_kind = 'organization'` AND `organization_id = :tenant`)) against a
  corpus whose artifacts do not yet carry the column.

This was defensible while V2 was a single-corpus read pipeline being built stage by stage.
It stops being defensible at Phase 15. **Why this decision has to land now, before Phase 15
rather than during it:**

1. **Tenancy is a property of the contracts, and the contracts are about to become load
   bearing.** Phase 14.5 stabilized `pipeline-contracts` precisely so that Missions, Agents,
   and Tools can be built on top of it. Every one of those introduces a *new* carrier of
   tenant context (a mission's owner, an agent's data scope, a tool's authorization). If we
   decide the model after they exist, we retrofit it into four subsystems instead of
   defining it for one.
2. **Retrofitting tenancy is the failure mode that produces cross-tenant leaks.** A tenant
   filter added late is a filter someone can forget to apply. The only version of this that
   holds is one where the tenant is *structurally impossible to omit* — a required field on
   the contract that enters the pipeline, not an argument a caller remembers to pass.
3. **The audit trail is now complete (Phase 14.5) and is about to be persisted.** An
   `AuditRecord` without a tenant cannot be stored in a tenant-scoped, append-only log — and
   an audit record retrofitted with a tenant later cannot vouch for the runs written before
   it. The record's shape must be right *before* a sink exists.
4. **Phase 15 will introduce the first consequential actions.** Tools with side effects and
   human gates are exactly where a tenancy mistake stops being a data-exposure bug and
   becomes an unauthorized *change* to a customer's compliance posture.

This ADR therefore fixes the model, the vocabulary, and the invariants **now**, and builds
none of it. That split is deliberate: Phase 14.5 is a stabilization phase, and shipping
half a tenancy implementation would be worse than shipping none — a partial tenant filter
reads as a guarantee it does not provide.

## Decision

### 1. What a Tenant is

**A Tenant is the organization that owns data, work, and decisions on the platform — the
unit of isolation, authorization, residency, and billing. It is the customer, not a user, a
workspace, a project, or a deployment.**

- **Identity.** A tenant has a stable, opaque `tenant_id`, generated by the platform and
  never derived from anything mutable or guessable (not a name, not a domain, not an email).
- **Granularity.** One tenant = one contracting organization. Users, workspaces, and
  missions belong to exactly one tenant. A user who legitimately works for two organizations
  has one identity and two memberships, and acts *within one tenant at a time* — never
  across both in a single request.
- **`tenant_id` is `organization_id`.** V2's existing knowledge model already scopes rows by
  `organization_id`; these are the same identifier under two names, and new code uses
  `tenant_id`. (The `apps/web` V3 organization work established the same equivalence.)
- **What a tenant is *not*.** It is not a data-partitioning hint, not a UI grouping, not
  optional metadata, and not something the platform may infer. Absent tenant context is an
  error, never a default and never "all tenants."

### 2. The two-library scope (the one exception, made explicit)

Not all knowledge is owned by a tenant. A framework, a law, and a standard are **`GLOBAL`**:
shared, identical for everyone, owned by no one. A tenant's policies, evidence, and
documents are **`ORGANIZATION`**: theirs alone.

**Every read spans `GLOBAL` ∪ `ORGANIZATION(tenant_id)`, and nothing else.** This is the
*only* way a query ever sees a row it does not own, it is a fixed rule rather than a
per-caller choice, and it is one-directional: global knowledge is readable by every tenant
and writable by none of them through the pipeline. Cross-*organization* reads do not exist
at any scope, for any caller, under any role.

### 3. Where tenant context enters the platform

**Tenant context enters at exactly one place: the authenticated request boundary, from the
verified identity — never from the request body, a query parameter, a header a client
controls, an LLM output, or a retrieved document.**

```
  OIDC/SSO identity  ─→  [ AUTH BOUNDARY ]  ─→  TenantContext  ─→  the pipeline
  (subject, org claim)    resolve + verify        (immutable)        (carried, never derived)
        │                  membership
        │
   ✗ request body     ✗ query param     ✗ client header     ✗ LLM output     ✗ document content
```

The boundary resolves the caller's identity to a tenant, verifies the membership, and mints
an immutable **`TenantContext`** value object. Everything downstream *carries* it. Nothing
downstream *decides* it, widens it, or re-derives it.

`TenantContext` is the contract (shape fixed here; construction is Phase 15+):

| Field | Meaning |
|---|---|
| `tenant_id` | The organization. Required. Never empty, never a wildcard. |
| `principal_id` | The authenticated actor within that tenant (a user, a service, or an agent acting on their behalf). |
| `roles` | The principal's roles, for RBAC/ABAC decisions made at the boundary and re-checked at side-effect boundaries. |
| `region` | The data-residency region the request is bound to (CLAUDE.md §20). |

**Invariants.**
- **No default tenant.** There is no "system tenant," no fallback, and no `tenant_id=None`
  path that means "everything." A missing or unresolvable tenant fails the request closed.
- **Immutable for the life of a run.** No stage may replace, widen, or re-derive it. A run
  serves one tenant, start to finish.
- **Untrusted input can never name a tenant.** Model output and retrieved document content
  are data (CLAUDE.md §19). A `tenant_id` appearing in either is a prompt-injection attempt,
  not context — the pipeline reads tenant only from `TenantContext`.

### 4. How tenant context flows through the pipeline

**`TenantContext` becomes a required field on `UserRequest` — the pipeline's entry contract
— and is threaded down the existing chain rather than passed alongside it.** Making it part
of the contract, not a parameter, is what makes omission a construction error rather than a
review catch.

```
UserRequest(tenant)                          ← the only place tenant enters
  → DecisionEngine      plan carries tenant       (routing may not vary tenant scope)
  → RetrievalEngine     Filter.scope = GLOBAL ∪ ORGANIZATION(tenant)   ← enforced IN the provider
  → ContextBuilder      every block traces to a source that passed the scope
  → PromptOrchestrator  prompt is built only from scope-checked context
  → GenerationEngine    request carries no tenant identifiers into the model
  → Answer / PipelineResult / AuditRecord (tenant-stamped)
```

Stage rules, in the language of the engines that exist today:

- **Decision.** The plan carries the tenant through. Classification and routing are tenant-
  *blind* by design: no intent, workflow, or budget may depend on which tenant is asking,
  so no routing bug can widen a scope.
- **Retrieval.** `Filter` gains the tenant scope, and — per the existing "filter inside the
  provider, never post-hoc" rule — the scope predicate is applied *inside* each provider,
  first, as part of the query that reaches the store. A post-hoc filter over cross-tenant
  candidates is forbidden: it would mean untrusted rows were read, ranked, and held in
  memory before being dropped. Every candidate's scope is re-verified at the engine boundary
  (defence in depth), and an out-of-scope chunk is a **failure**, not a filtered result.
- **Context.** No new enforcement: every block descends from a chunk that already passed the
  scope gate, and citations preserve the provenance that proves it.
- **Prompt.** Tenant identifiers are **not** rendered into prompts. The tenant governs
  *which* evidence is assembled; it is not a fact the model needs, and putting an opaque id
  in a prompt only creates a way to leak or hallucinate one.
- **Generation.** The provider port stays tenant-agnostic. Tenancy is decided before the
  call, never inside it; the LLM has no authority over scope.
- **Result & audit.** `PipelineResult` and `AuditRecord` are stamped with the tenant that
  ran (see §6).

**Caching, memory, and any future derived state are keyed by `tenant_id` as part of the
key** — never as a filter over shared results. An unkeyed cache is a cross-tenant leak with
extra steps.

### 5. Forward compatibility: Missions, Agents, Tools

These do not exist yet. The tenancy contract they will be built against is fixed here, so
they are born tenant-scoped rather than retrofitted.

- **Missions (CLAUDE.md §8).** A Mission is tenant-owned at creation: `tenant_id` is bound
  when it is created and is immutable for its life — a mission cannot change tenant, and a
  replanned or resumed mission re-enters with the same `TenantContext` it started with. It
  is part of the mission's identity, not its metadata. Mission memory, state, plans, and
  step outputs are tenant-scoped storage. A mission may only spawn sub-missions in its own
  tenant. Resumption after a human gate re-verifies that the approver belongs to the
  mission's tenant — a gate approved by an outsider is not a gate.
- **Agents (CLAUDE.md §11).** An agent is a reasoning unit with no tenant of its own; it
  *borrows* the `TenantContext` of the mission step it is executing, and its data and tool
  access are least-privilege *within* that tenant. An agent cannot widen its own scope, and
  the Orchestrator — not the agent, and never the LLM — supplies the context. An agent that
  could name its own tenant would be an agent that could be prompt-injected into another
  one's data.
- **Tools (CLAUDE.md §9/§10).** `TenantContext` is a **required** part of every Tool
  invocation — this is already written into the Tool contract in CLAUDE.md §9, and this ADR
  fixes its shape. It holds identically for all six callers (Orchestrator, API, UI, Workflow
  Engine, Scheduled Jobs, Tests): a Tool called from a cron job needs a tenant exactly as
  much as one called from the UI, and a scheduled job that "runs for all tenants" is N
  single-tenant invocations, not one wildcard invocation. The Registry enforces which
  roles/tenants may call which Tools; idempotency keys are scoped per tenant so two tenants
  cannot collide; and a Tool's declared side-effect profile is evaluated within its tenant.

### 6. Forward compatibility: Audit and Events

- **Audit (CLAUDE.md §19; ADR 0015).** `AuditRecord` gains `tenant_id` as a required field.
  The Phase 14.5 record is complete in *what it captured*; it is not yet complete in *who it
  belongs to*, and that is the last thing missing before an `AuditSink` can persist. The
  rules the sink will be built against: the trail is **append-only and tenant-scoped**; an
  auditor for tenant A can never read tenant B's records; a record's tenant is stamped at
  finalization from the run's `TenantContext` and is never inferred; retention and residency
  follow the tenant's region.
- **Events (CLAUDE.md §16; ADR 0009).** Every `DomainEvent` carries `tenant_id`, on the base
  class — so it is structurally impossible to publish an event without one, exactly as
  `trace_id` is today. Subscribers are tenant-scoped: a handler receives only events for
  tenants it is authorized for, and a wildcard subscriber (an audit sink, a tracer) is
  authorized platform-wide *precisely because* it is scoped by construction rather than by
  filtering. This holds for the in-process bus today and for any future transport: an event
  stream partitioned by tenant is a requirement of the port, not a property of one
  implementation.

### 7. What this ADR explicitly does not do

No tenancy is implemented in Phase 14.5. No contract gains a `tenant_id` field, no filter
gains a scope predicate, no event gains a tenant, no `TenantContext` type is created. This
ADR is the contract and the governance; the build is Phase 15+.

The implementation, whenever it lands, is **not** additive-and-optional the way Phase 13's
hardening was. An optional tenant is not a tenant. `TenantContext` arrives as a required
field, and every call site is updated in the same change.

## Consequences

**Positive**
- The model is decided before Missions, Agents, and Tools exist to retrofit it into — the
  cheapest moment it will ever be decided at, and the last one before it gets expensive.
- Tenancy becomes a *contract* property: omitting a tenant fails at construction, not at
  review, and not in production.
- The `AuditRecord`'s shape is settled before a persistence layer commits to it, so the
  first persisted record is already audit-legal.
- Reviewers get a concrete rule to reject changes against, replacing CLAUDE.md §20's
  principle with an enforceable specification.
- The two-library exception is written down as the *only* cross-scope read, so any other
  one is unambiguously a bug.

**Negative / costs**
- The gap between the stated principle and the code stays open until Phase 15. This ADR
  makes the gap explicit and bounded rather than closing it; anyone reading CLAUDE.md §20
  against the V2 code must know that today's V2 pipeline has **no** tenant enforcement and is
  not fit to serve two tenants' data. It is single-corpus by construction, and that is the
  only thing currently keeping it safe.
- Making `TenantContext` required is a breaking change to `UserRequest` and every caller —
  deliberately taken over an optional field that would be silently omitted.
- Tenant-scoped cache keys reduce hit rates versus a shared cache. Accepted without argument:
  a shared cache across tenants is a leak.
- Threading context through every stage adds boilerplate to signatures that read fine today.

## Alternatives considered

- **Implement tenancy now, inside Phase 14.5.** Rejected: the phase is explicitly
  stabilization, and its constraints forbid it. More substantively, tenancy touches every
  contract and every engine — it is the opposite of a stabilizing change, and bundling it
  with the audit and citation work would make all three harder to review.
- **Defer the decision to Phase 15 and decide it alongside Missions.** Rejected: this is the
  retrofit path. Missions, Agents, and Tools each introduce a new carrier of tenant context;
  designing them against an undecided model guarantees three inconsistent answers and a
  fourth migration.
- **Add `tenant_id` as an optional field now, enforce later.** Rejected, and worth naming
  as the most tempting option: an optional tenant is a field every existing call site
  omits, so the pipeline would carry `None` everywhere while *looking* tenant-aware — the
  worst of both, and precisely the "convention, not construction" failure ADR 0014 rejected.
- **Tenant-per-deployment (one instance per customer).** Rejected: does not scale to
  thousands of tenants and abandons the shared global knowledge library, which is a core
  reason the two-library model exists. (Consistent with ADR 0014.)
- **Row-Level Security in Postgres as the sole mechanism.** Rejected *as the sole*
  mechanism: it protects one store, while tenancy must hold for events, caches, memory,
  missions, and tool calls too. It remains a strong defence-in-depth layer under the
  contract, not a substitute for it.
- **Derive the tenant from the request payload or a header.** Rejected: that is
  client-controlled input, which makes cross-tenant access a matter of editing a field.
  Tenant comes from the verified identity or the request fails.
